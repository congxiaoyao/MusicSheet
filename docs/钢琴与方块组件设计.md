## 一、背景

### 1.1 项目与练琴页

MusicSheet 是纯原生 TypeScript（零框架）的五线谱↔简谱编辑与播放工具。练琴页是规划的**第三级页面**（曲谱库 → 编辑器 → 练琴），是编辑完成后的主要练琴场景。

练琴页主视图是**音游式键位提示与纸质谱面的融合体**——不是并列，是叠加。屏幕底部是键盘，键盘上方是瀑布流方块（往下掉的音块），方块之上是铺满的乐谱。

```
┌──────────────────────────────────────────────┐
│ 顶栏（播放/变速/AB/单手/节拍器）              │
├──────────────────────────────────────────────┤
│  ▍当前行（清晰）                              │
│   （半透明）后续行    ← ScoreSheet 组件       │
│        ▓▓      ▓▓     ← 方块组件（本文档）    │
│   ━━━━━━━━━━━━━ 判定线                       │
│   🎹🎹🎹🎹🎹🎹🎹    ← 钢琴组件（本文档）       │
└──────────────────────────────────────────────┘
```

### 1.2 为什么这两个组件一起设计

钢琴组件和方块组件是练琴页里**耦合最紧**的两个组件，有三条硬耦合：

1. **共享坐标系**：方块的水平位置 = 键的位置。同一套坐标算出来，否则对不上。
2. **共享"活跃音"状态**：方块落到底部时，对应的键要点亮。
3. **共享左右手过滤**：单手隔离时，方块的隐藏和键盘高亮的隐藏必须同步。

分开设计会让这三条耦合变成隐式约定，迟早对不上。一起设计把耦合点显式定清楚。

### 1.3 解耦方案（关键设计决策）

经讨论确定：**两个组件不直接通信**，通过 PracticeApp（controller）间接同步。具体：

- **坐标函数 `midiToX`**：抽成独立纯函数，两组件各自调用，不互相依赖。只有一份坐标逻辑，两者必然对齐。
- **命中时机各自独立算**：方块按"接近判定线"算自己的命中效果，键盘按"当前正在响的音"算点灯。两者不要求完全同步，各自在自己的职责里最准。
- **数据来源分离**：方块拿完整乐谱自己算未来音符；键盘不接触乐谱，controller 算"当前响的音集合"喂给它。

```
PracticeApp.onTick(beat):
  activeMidis = 从乐谱算当前响的音（原始 midi）
  keyboard.setActiveMidis(activeMidis)   ← 键盘自己映射指法后点灯
  waterfall.onTick(beat)                  ← 方块自己算位置和命中
```

### 1.4 组件模式

遵循项目的命令式工厂 + Handle 模式（详见 `src/ui/playback-card.ts`）：

```ts
export function buildXxx(initial, cb): XxxHandle {
  // 建内部 DOM、绑事件
  return { el, onTick, /* 其他方法 */ };
}
```

组件不调用 App 方法，只通过 callbacks 向外报告。高频更新走细粒度命令式方法，不走全量 refresh。

## 二、公共坐标函数（先于两组件实现）

### 2.1 为什么是公共的

方块横轴 = 键盘位置。如果 `midiToX` 属于某个组件，另一个就得依赖它（时序耦合：键盘黑键 rAF 定位后 midiToX 才准）。抽成纯函数后，两者各自调用，**消除时序依赖**。

### 2.2 接口

```ts
// src/practice/key-coords.ts（或类似公共位置）

/** 键盘音域范围 */
export interface KeyRange { low: number; high: number; }

/**
 * midi → 键盘中心的横坐标百分比（0~100），相对"键区内容宽度"。
 * 白键：取自身中心
 * 黑键：位于左侧白键右沿（两白键交界处）
 */
export function midiToX(midi: number, range: KeyRange): number;

/** 该音域下的白键 midi 序列（供键盘构建和方块对齐复用） */
export function whiteKeys(range: KeyRange): number[];

/** 一个白键占键区的宽度百分比 = 100 / 白键数 */
export function whiteKeyWidth(range: KeyRange): number;

/** 一个黑键的宽度百分比 = 白键宽 × 0.6（与 playback-card 黑键等宽） */
export function blackKeyWidth(range: KeyRange): number;

/** 某 midi 的方块宽度百分比：白键→whiteKeyWidth，黑键→blackKeyWidth。
 *  方块和对应键等宽，落下来正好"落进"键里（参考 Synthesia 主流做法）。 */
export function noteWidth(midi: number, range: KeyRange): number;
```

### 2.3 坐标系基准（关键）

所有百分比都相对**键区内容宽度**（白键实际排列的区域），不含键盘容器的 padding。

现有 `playback-card.ts` 的坑：keyboard 容器有 `padding: 6px`，CSS 百分比相对 padding box（含 padding），导致黑键用 `百分比等分` 定位时累积偏差越往外越大——所以 playback-card 才用 rAF 读 DOM 真实边界来兜底。

公共纯函数的解法：**不依赖 DOM，纯按白键数等分算坐标**。键盘组件渲染时，让白键在 padding 内部的 content box 里严格 flex 等分（`box-sizing: border-box`，白键 flex basis = `whiteKeyWidth()`），黑键用 `midiToX` 定位。这样纯函数和 DOM 渲染用**同一套数学**，必然一致，不需要 rAF 兜底。

> 这是 playback-card 的遗留问题在新组件里的修正机会。playback-card 不改（编辑器在用），新键盘组件用纯函数坐标系，从根上消除偏差。

### 2.4 验证标准

键盘渲染后，每个键的中心 x 和 `midiToX(midi, range)` 的结果必须一致（误差 < 1px）。方块宽度 `noteWidth` 和对应键的实际宽度一致。这是方块能对齐键盘的前提。

## 三、钢琴组件（Keyboard）

### 3.1 职责

1. **渲染键盘**：按音域渲染白键 + 黑键（flex 等分 + 黑键精确定位）
2. **键面标注**：音名 / 唱名 / 八度，三选一可切
3. **高亮当前音**：controller 传入"当前响的原始 midi 集合"，键盘自己根据指法模式映射后点亮
4. **指法模式**：cfixed（固定 C 调，配合电钢琴移调）/ follow（跟随调号，真实音高），完整实现
5. **键宽/音域动态调整**：键盘左右边缘拖把柄，连续控制键宽（键数自动算），方块横轴自动跟随对齐
6. **高度动态调整**：键盘上边缘拖把柄，拖动改高度

### 3.2 键盘宽度与键宽（关键）

**键盘整体宽度 = 铺满练琴页容器宽度**（响应式，`width: 100%`）。键盘在屏幕底部占一整行。

**白键 flex 等分**：所有白键 `flex: 1` 等分容器宽度，每个白键宽 = `容器宽 / 白键数`。`box-sizing: border-box` 保证 border 计入宽度、白键严格等宽。

**黑键宽度**：白键宽的 **0.6**（约 60%，真实钢琴比例）。黑键绝对定位在白键交界处（中心 = 左侧白键右沿）。

**音域决定键数、键数决定键宽**：
- 音域越宽（如 C2-C6，29 个白键）→ 每个键越窄 → 键面标注越挤
- 音域越窄（如 C4-C5，8 个白键）→ 每个键越宽 → 标注宽松
- 默认按谱面音域自动确定（曲子用到的音 + 少量上下冗余），用户可手动调

**用纯函数坐标系，不用 rAF 兜底**：与 playback-card 不同，新键盘组件用 §2 的公共纯函数（`midiToX`/`whiteKeyWidth`）算坐标，白键 flex basis 直接设 `whiteKeyWidth()`，黑键 `left` 直接设 `midiToX()`。纯数学，不读 DOM，从根上消除 playback-card 因 padding 导致的百分比偏差（见 §2.3）。

> playback-card 的 `buildKeyboard` 用 rAF 读 DOM 真实边界定位黑键，是因为它用 CSS 百分比等分时 padding 会累积偏差。新组件改用纯函数坐标系后不需要 rAF。

### 3.3 音域/键宽动态调整（拖把柄，连续控制键宽）

键盘左右边缘各一个拖把柄（半透明竖条，hover 显现），拖动连续改变键宽：

**交互模型：控制键宽，键数自动算**
- 用户拖动控制的是**每个白键的宽度**（px，连续值）
- 键数 = `floor(容器宽 / 白键宽)`，自动派生
- 键宽连续变化 → 键数到阈值跳变（多/少一个键），但跳变时键宽视觉几乎无感（如 800px 容器，8→9 键时键宽从 ~89px 变 ~88.8px）
- 拖右边缘 = 改变右端音域（高音延伸/收缩），拖左边缘 = 改变左端音域（低音延伸/收缩）

**实时重排，不闪烁**
- 拖动每帧：用纯函数重算所有键的 `left`/`width`（`midiToX` + `whiteKeyWidth`），DOM 只更新 style 不重建元素 → 连续平滑、无闪烁
- 键数变化时才增删键元素（append/remove 单个白键 DOM + 对应黑键），不整体重建
- 方块横轴用同一套纯函数，自动跟随对齐

**防高亮丢失**：拖动中如果键元素被增删，`keyElByMidi` 映射要同步更新；播放中的高亮状态（activeMidis）在重排后重新应用，不丢。

**上下限约束**：
- 键宽下限：不能窄到键面标注看不清（如最小 28px）
- 键宽上限：不能宽到不合理（如最大 80px）
- 音域下限：不能少于曲子用到的音域（否则该亮的键显示不出来）
- 音域上限：不超过合理范围（如最多 88 键全钢琴）

**拖动中可播放**：拖把柄不影响播放，播放中也能调。

**持久化**：键宽（或等价的音域）调整结果存 localStorage，下次保持。

### 3.4 高度动态调整（拖上边缘）

键盘上边缘一个横向拖把柄，拖动改高度：

- **向上拖** = 键盘变高（键面标注更清楚）
- **向下拖** = 键盘变矮（给谱面/方块区让空间）
- **连续拖**（不像音域那样阶梯式），高度是连续值
- **联动方块区**：键盘高度变 → 判定线位置变 → 方块区有效高度变。controller 收到 `onHeightChange` 后调 `waterfall.setBounds` 更新方块区下边界

**上下限约束**：
- 下限：不能矮到键面标注看不清（如最小 80px）
- 上限：不能高到挤压谱面太多（如最大不超过视口 40%）

**记忆**：高度和音域的调整结果应持久化（localStorage），下次进练琴页保持。

### 3.5 完整复刻 playback-card 的键盘能力

**不是简化版**。以下能力从 `src/ui/playback-card.ts` 原样抽出：

| 能力 | 来源 | 说明 |
|---|---|---|
| 白键/黑键构建 | `buildKeyboard()` L317-373 | flex 等分 + 纯函数定位（替代原 rAF，见 §3.2） |
| 键面标注（音名/唱名/八度） | L331-346 | 三选一，复用 `midiName`/`midiSolfege` |
| 高亮切换（切 class 不重建） | `updateKeyboardHighlight()` L400-408 | 高频零闪烁 |
| 指法模式映射 | `highlightMidi()`（被 computeActiveMidis 调用） | cfixed/follow |
| 键盘重建签名（音域/标注变才重建） | `keyboardSig` L313,371 | 避免播放时重建闪烁 |
| 中央 C 标记 | L330 | 视觉定位参考 |

### 3.6 指法映射在组件内部

controller 传入的是**原始 midi**（乐谱里的真实音高）。键盘组件自己根据当前指法模式映射成要点亮的键：

- `follow` 模式：直接用原始 midi 点亮对应键
- `cfixed` 模式：简谱 1-7 映射到 C 调白键（配合电钢琴移调），原始音经映射后点亮不同键

这样指法逻辑完全封装在键盘组件内，controller 不用管。参考 `playback-card.ts` 现成的 `highlightMidi` 函数。

### 3.7 接口设计

```ts
export interface KeyboardInitial {
  range: KeyRange;                  // 音域（默认按谱面自动，用户可拖把柄调）
  height: number;                   // 键盘高度 px（用户可拖上边缘调）
  labels: 'name' | 'solfege' | 'octave' | 'none';  // 键面标注
  fingering: 'cfixed' | 'follow';   // 指法模式
  key: Key;                         // 调号（唱名/指法映射用）
}

export interface KeyboardCallbacks {
  /** 用户拖把柄改了音域 → controller 据此更新方块的 range（midiToX 重算） */
  onRangeChange?: (range: KeyRange) => void;
  /** 用户拖上边缘改了高度 → controller 据此调 waterfall.setBounds（判定线移动） */
  onHeightChange?: (height: number) => void;
  onLabelChange?: (labels: string) => void;
  onFingeringChange?: (f: string) => void;
}

export interface KeyboardHandle {
  el: HTMLElement;
  /** controller 每帧调：传入当前响的原始 midi 集合。键盘内部做指法映射后点灯。 */
  setActiveMidis(midis: number[]): void;
  setRange(range: KeyRange): void;
  setHeight(height: number): void;
  setLabels(labels: string): void;
  setFingering(f: string): void;
  /** 清除所有高亮（停止/暂停时） */
  clearHighlight(): void;
}
```

> 音域和高度的调整结果应持久化（localStorage），下次进练琴页保持。controller 负责存取，键盘组件只报告变化。

### 3.8 音域默认值

默认按谱面音域自动确定（曲子用到哪些音就显示哪些键）。参考 `playback-card.ts` 的 `whiteKeyRange`。支持用户手动调整范围。

## 四、方块组件（Waterfall）

### 4.1 职责

1. **渲染方块**：每个音符一个方块，从当前行底部向下掉落
2. **横轴定位**：方块水平位置 = `midiToX` 算出的键盘位置
3. **下落动画**：接收 beat 驱动，算每个方块的 y 坐标
4. **命中效果**：接近判定线时加 active 视觉
5. **左右手过滤**：单手隔离时隐藏一只手的方块
6. **始终显示音名**：方块上标注音名

### 4.2 数据来源

方块组件**拿完整乐谱**（Score），自己解析出每个音符的位置信息（音高、起始拍、时值、左右手）。它需要算"未来 N 拍内的音"以提前显示下落的块。

```ts
// 方块从乐谱解析出的内部数据结构
interface FallNote {
  midi: number;
  beat: number;       // 起始拍
  duration: number;   // 时值（拍）
  hand: 'R' | 'L';    // 左右手（treble=R, bass=L）
}
```

### 4.3 方块宽度 = 对应键宽（关键）

**方块宽度等于它要按的那个键的宽度**——白键的方块 = 白键宽，黑键的方块 = 黑键宽（更窄）。这是 Synthesia/SeeMusic/Piano VFX 等主流工具的标准做法。

理由：
1. **空间精确映射**：方块"落进"它对应的键里，用户看方块落哪就按哪，不用解码颜色/标签
2. **强化键盘交替模式**：黑白方块交替的视觉帮助辨认升降记号段落
3. **1:1 对应**：方块就是键的投影，物理直觉

**实现**：方块宽度用公共纯函数 `noteWidth(midi, range)`（§2.2）算，白键返回 `whiteKeyWidth`、黑键返回 `blackKeyWidth`。方块的 `left` 用 `midiToX`（居中），`width` 用 `noteWidth`。两者都基于同一套坐标系，天然和键盘对齐。

**比例微调**：方块宽度可取对应键宽的 90%~100%（略窄一点留视觉间隙），但不区分"统一宽度"——白键方块就是比黑键方块宽，和真实键宽一致。

> 这是原型 practice-prototype.html 的修正：原型方块宽度写死 28px，固定像素 vs 响应式键宽导致宽屏对不齐。改成百分比后随容器等比缩放。

### 4.4 下落与命中（各自独立算）

- **下落**：每个方块的 y = `判定线y - 块高 - (note.beat - beat) × PX_PER_BEAT`。未来音（beat 差为正）在判定线上方，过去音（差为负）在下方淡出。
- **命中效果**：方块接近判定线时（`|note.beat - beat| < 阈值`）加 `active` class（放大/光环）。**方块自己判断**，不依赖键盘。
- **判定线 = 键盘上沿**：方块底边贴到键盘上沿时，就是该按的时刻。

### 4.5 掉落区域（裁剪）

方块只在"键盘正上方的固定区域"可见，超出区域顶部裁掉。区域上边界由 ScoreSheet 的当前行底部决定（通过 PracticeApp 协调，见 ScoreSheet 文档的 `onLineLayout` 回调）。

- 区域顶部 fade（mask 渐变）：方块接近上边缘时自然淡出，不生硬
- 区域高度 = ScoreSheet 当前行底部 到 键盘顶部

### 4.6 方块内容

- **始终显示音名**（如 C4、G♯4），初学者降低认知负担
- **左右手颜色区分**：右手蓝、左手橙（与键盘高亮颜色一致）
- 方块高度按时值：长音块更高，短音块更矮

### 4.7 接口设计

```ts
export interface WaterfallInitial {
  notes: FallNote[];           // 从乐谱解析出的音符（或直接传 Score 内部解析）
  range: KeyRange;             // 音域（传给 midiToX 用）
}

export interface WaterfallCallbacks {
  // 方块组件一般不需要向外回调（它只接收驱动、自己渲染）
  // 除非要支持"点击方块跳转"之类的交互（可选，后续）
}

export interface WaterfallHandle {
  el: HTMLElement;
  /** controller 每帧调：驱动方块下落 + 命中判断 */
  onTick(beat: number): void;
  /** 乐谱变更时重新解析音符 */
  setNotes(notes: FallNote[]): void;
  /** 音域变化时重新算横轴 */
  setRange(range: KeyRange): void;
  /** 单手隔离：'both' | 'R' | 'L' */
  setHandFilter(hand: 'both' | 'R' | 'L'): void;
  /** 设置掉落区域上边界（ScoreSheet 当前行底部） */
  setBounds(info: { topY: number; bottomY: number }): void;
}
```

## 五、两者的协作（通过 PracticeApp）

```
PracticeApp（controller）持有三者：
  - scoreSheet: ScoreSheetHandle
  - keyboard: KeyboardHandle
  - waterfall: WaterfallHandle

初始化：
  keyboard = buildKeyboard({ range: autoRange(score), ... }, cb)
  waterfall = buildWaterfall({ notes: parseFallNotes(score), range }, cb)
  // midiToX 是公共纯函数，两者各自 import，不互相传

ScoreSheet.onLineLayout 回调：
  → waterfall.setBounds({ topY: lineBottomY, bottomY: keysTop })

PracticeApp.onTick(beat):
  activeMidis = computeActiveMidis(beat, score)   // 算当前响的原始 midi
  scoreSheet.onTick(beat)     // 谱面：行滚动 + 符头高亮
  waterfall.onTick(beat)      // 方块：位置 + 命中
  keyboard.setActiveMidis(activeMidis)  // 键盘：点灯

单手隔离（用户切 handFilter）：
  waterfall.setHandFilter(hand)
  keyboard 高亮时过滤（controller 传 midi 时按 hand 过滤，或键盘自己过滤）
```

> NoteApp、键盘、方块的完整装配是后续 plan 的范围。本文档只定义两个组件的接口和职责。

## 九、关键风险

1. **纯函数坐标系的正确性**：`midiToX`/`noteWidth` 等纯函数是新键盘和方块的共同基准。键盘渲染（白键 flex basis、黑键 left/width）必须和纯函数用同一套数学，否则方块对不齐键盘。关键是键盘 padding 不进坐标系——百分比相对 content box，白键在 padding 内严格等分。
2. **方块宽度随键宽等比缩放**：方块宽度用 `noteWidth`（百分比），不能写死像素。不同屏幕宽度下键宽变化，方块要跟着变才能保持对齐。
3. **指法映射的完整移植**：cfixed/follow 逻辑要从 playback-card 完整搬，不能简化。参考 `highlightMidi` 函数。
4. **方块区域与 ScoreSheet 的协调**：方块上边界依赖 ScoreSheet 的当前行底部（动态值）。setBounds 要在 ScoreSheet onLineLayout 触发时及时更新，否则方块会盖到当前行。
5. **三档谱面切换时的行高变化**：ScoreSheet 切 mode（五线/简谱/对照）时行高变，方块上边界要跟着变。onLineLayout 回调要能正确反映新行高。
6. **和弦音**：多个音同时响（和弦）时，方块要并排显示多个块，键盘要点亮多个键。computeActiveMidis 已处理和弦组（playback-card L384-388），方块解析时也要保留和弦信息。
7. **拖把柄连续改键宽时的重排平滑**：用户连续拖动改键宽，每帧用纯函数重算所有键的 left/width（只更新 style，不重建 DOM）。键数变化时才增删单个键元素，不整体重建。关键是拖动中 `keyElByMidi` 映射同步更新 + 高亮状态（activeMidis）重排后重新应用，不丢不闪。
8. **拖把柄改高度时与方块区联动**：键盘高度变 → 判定线位置变 → controller 收到 `onHeightChange` 后必须及时调 `waterfall.setBounds` 更新方块区下边界，否则方块落点和键盘脱节。
9. **音域/高度持久化**：调整结果存 localStorage，controller 负责存取，键盘组件只报告变化（onRangeChange/onHeightChange）。
